## 🎯 設計原則

### フロントエンド優先戦略
1. **UI設計を最優先**: ユーザー体験から設計を開始
2. **型安全性**: TypeScriptでDB型定義を先に確定
3. **段階的移行**: モックデータ → リアルデータへスムーズに移行
4. **コンポーネント再利用**: 共通コンポーネントの設計を重視

### データベース設計
1. **正規化**: 第3正規形を基本とする
2. **拡張性**: Phase 2, 3 への拡張を考慮
3. **セキュリティ**: RLSポリシーで細かくアクセス制御
4. **パフォーマンス**: 適切なインデックス設計

### API設計
1. **RESTful**: リソース指向の設計
2. **一貫性**: 命名規則、エラーフォーマットの統一
3. **バージョニング**: 将来的なAPI変更に対応
4. **ドキュメント**: OpenAPI仕様での記述

## 📝 作成順序

1. **要件定義** → 完了済み（`requirement.md`）
2. **画面遷移図** → ユーザーフローを明確化
3. **UI/UX設計書** → デザインシステムと画面レイアウト
4. **データベース設計書** → データモデルと型定義
5. **API仕様書** → フロントエンドとバックエンドのインターフェース
6. **モックデータ仕様書** → 開発・テスト用データ

## 🔄 更新ルール

- 設計変更時は必ずドキュメントを更新
- 変更履歴をドキュメント内に記録
- レビューを経てから実装に反映
